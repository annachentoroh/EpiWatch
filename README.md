# EpiWatch — Інформаційно-аналітична платформа моніторингу епідеміологічної ситуації / Information and analytical platform for monitoring the epidemiological situation in the world and Ukraine.

**EpiWatch** — це комплексна інформаційно-аналітична вебплатформа, призначена для оперативної агрегації, геопросторового моніторингу та детального аналізу епідеміологічної ситуації як в Україні, так і на глобальному рівні. Проєкт розроблено студентами 2-го курсу Національного університету «Києво-Могилянська академія» (НаУКМА) факультету інформатики спеціальності 121 «Інженерія програмного забезпечення» відповідно до затверджених вимог Технічного завдання (ТЗ) та Статуту проєкту. 

Система розв'язує проблему «інфодемії» (поширення дезінформації навколо біологічних загроз, таких як COVID-19, хантавіруси, спалахи кору тощо) шляхом надання користувачам верифікованих статистичних маркерів, інтерактивного картографічного шару, аналітичних інструментів та освітніх модулів, що базуються на офіційних стандартах ВООЗ (WHO) та Global Health Observatory (GHO).

## 🌎 Загальні відомості та призначення

### Обґрунтування актуальності
Динамічний характер виникнення нових штамів патогенів вимагає наявності публічних цифрових систем, здатних швидко інформувати населення. **EpiWatch** поєднує в собі математичну точність візуалізації статистичних даних та доступність викладу для цивільних осіб.

### Цільова аудиторія платформи:
* **Громадяни (Загальні користувачі):** Оцінка ризиків у регіоні перебування, самоскринінг первинних ознак нездужання, перегляд пам'яток першої допомоги.
* **Медичні аналітики та дослідники:** Моніторинг динаміки приросту захворюваності, експорт статистичних зрізів для розрахунку показників летальності (CFR) та індексу репродукції ($R_0$).
* **Освітні та громадські організації:** Отримання науково обґрунтованих спростувань псевдомедичних теорій з посиланнями на першоджерела.


## ⚡ Функціональна архітектура (Ключові модулі)

### 1. Модуль геопросторового моніторингу (Інтерактивна мапа)
* **Технологічна основа:** Бібліотека `Leaflet.js` з картографічною підкладкою `OpenStreetMap`.
* **Динамічний рендеринг маркерів:** Візуалізація вогнищ інфекцій за координатами (широта/довгота). Колір маркера динамічно змінюється залежно від епідеміологічної небезпеки захворювання та кількості активних випадків:
  * 🟢 **Низький рівень загрози:** Локальні контрольовані випадки.
  * 🟡 **Середній рівень загрози:** Регіональне поширення, тенденція до зростання.
  * 🔴 **Високий / Критичний рівень (Епідемія):** Масштабні спалахи, що вимагають карантинних заходів.
* **Інформаційні балуни (Popups):** Натискання на маркер відкриває модальне вікно з інтегрованою експрес-метрикою: кількість підтверджених випадків, одужань, летальних наслідків та дата останньої синхронізації.

### 2. Аналітичний модуль (Графіки динаміки)
* **Технологічна основа:** `Chart.js` для рендерингу на HTML5 Canvas елементах.
* **Візуалізація трендів:** Побудова лінійних діаграм часових рядів та стовпчикових діаграм для порівняльного аналізу швидкості розповсюдження вірусів.
* **Фільтрація:** Можливість накладання декількох кривих інфекцій на одну координатну сітку для визначення математичної кореляції спалахів.

### 3. Модуль інтелектуального скринінгу (Пошук за симптомами)
* **Алгоритм підбору:** Інтерфейс надає інтерактивну систему чекбоксів (лихоманка, сухий кашель, задишка, висип тощо). Асинхронний алгоритм аналізує комбінаторні зв'язки симптомів та повертає список найбільш імовірних захворювань з відсотковим індикатором відповідності.
* **Попередження:** Модуль забезпечений жорстким дисклеймером про суто рекомендаційний характер оцінки та обов'язковість звернення до сімейного лікаря.

### 4. Модуль верифікації інформації («Антиміф») та Глосарій
* **Спростування фейків:** База даних структурованих карток, де кожному популістському міфу протиставлено розгорнуте академічне пояснення та пряме клікабельне URL-посилання на рецензоване джерело (ВООЗ, CDC).
* **Епідеміологічний глосарій:** Інформаційний блок із визначеннями ключових термінів: *Інкубаційний період*, *Летальність (CFR)*, *Зоонозні хвороби*, *Вектор (переносник)*, *Суперрозповсюджувачі*.


## 🛠️ Технічний стек та системні вимоги

### Клієнт-серверна модель
Проєкт розроблено з використанням модульної архітектури, де клієнтська частина повністю ізольована від прямого доступу до даних та взаємодіє з сервером виключно через асинхронні `Fetch API` запити до RESTful маршрутів.

```text
+-------------------------------------------------------------+
|                        КЛІЄНТ (UI)                          |
|  HTML5 / CSS3 (Grid, Flex, Custom Properties)               |
|  Vanilla JavaScript (ES6+ Modules)                          |
|  Внутрішні модулі: app.js, map.js, charts.js                |
+------------------------------+------------------------------+
                               |
                   HTTP REST Requests (JSON)
                               |
                               v
+-------------------------------------------------------------+
|                        СЕРВЕР (API)                         |
|  Node.js / Express.js Runtime Environment                   |
|  Модулі: server.js (Маршрутизація, CORS, Middleware)        |
|          scraper.js (Модуль автоматичного парсингу)         |
+------------------------------+------------------------------+
                               |
                      SQL / SQLite Driver
                               |
                               v
+-------------------------------------------------------------+
|                         БАЗА ДАНИХ                          |
|  SQLite3 (Локальний файловий рушій `epiwatch.sqlite`)       |
+-------------------------------------------------------------+
```

### Специфікація технологій:
* **Frontend:** Чистий JavaScript (без важких фреймворків як React/Angular) задля досягнення максимального показника First Contentful Paint (FCP). Семантичний HTML5, CSS3-стилі з повною адаптивністю (Responsive Design).
* **Backend:** Node.js, експрес-маршрутизатор Express, пакет `sqlite3` для взаємодії з СУБД.
* **Синхронізація:** `scraper.js` — скрипт автоматичного збору та парсингу оновлень епідеміологічних даних із відкритих API-інтерфейсів.

### Системні вимоги:
* **Для кінцевого користувача (Клієнт):**
  * Браузер: Chrome 100+, Firefox 100+, Edge 100+, Safari 15+.
  * Роздільна здатність екрана: Адаптивна верстка під мобільні пристрої (від 380px по ширині) та десктопні монітори (від 1280×720px).
  * Інтернет-канал: від 5 Мбіт/с для плавної передачі та рендерингу картографічних шарів Leaflet та Canvas-анімацій Chart.js.
* **Для розгортання (Сервер):**
  * Node.js: версія LTS v16.x або вище.
  * NPM: версія 8.x або вище.

## 🗄️ Проектування Бази даних (Схема SQLite)

Дані зберігаються у реляційній структурі в локальному файлі бази даних `epiwatch.sqlite`. База складається з трьох ключових таблиць, оптимізованих для швидкого читання за допомогою індексів.

### 1. Таблиця `diseases` (Реєстр нозологій)
Містить нормативно-довідкову інформацію про інфекційні хвороби.
| Назва поля | Тип даних | Обмеження (Constraints) | Опис призначення |
| :--- | :--- | :--- | :--- |
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | Унікальний внутрішній сурогатний ключ |
| `name` | TEXT | NOT NULL UNIQUE | Офіційна назва захворювання |
| `type` | TEXT | NOT NULL | Етіологічний тип (вірусне, бактеріальне, грибкове) |
| `description`| TEXT | NOT NULL | Детальний клінічний опис патогенезу |
| `incubation_period` | TEXT | — | Середня тривалість інкубаційного періоду |
| `danger_level` | TEXT | CHECK(`danger_level` IN ('Low', 'Medium', 'High')) | Рівень епідеміологічної небезпеки |

### 2. Таблиця `epidemiology_data` (Статистичні зрізи спалахів)
Зберігає геопросторові та кількісні показники зафіксованих епідеміологічних подій.
| Назва поля | Тип даних | Обмеження (Constraints) | Опис призначення |
| :--- | :--- | :--- | :--- |
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | Унікальний ідентифікатор запису події |
| `disease_id` | INTEGER | FOREIGN KEY REFERENCES `diseases`(`id`) | Зв'язок із таблицею реєстру хвороб |
| `region` | TEXT | NOT NULL | Географічна прив'язка (Країна/Область України) |
| `latitude` | REAL | NOT NULL | Географічна широта (WGS-84) для Leaflet |
| `longitude` | REAL | NOT NULL | Географічна довгота (WGS-84) для Leaflet |
| `confirmed_cases` | INTEGER | DEFAULT 0 | Кількість лабораторно підтверджених випадків |
| `recovered` | INTEGER | DEFAULT 0 | Кількість задокументованих випадків одужання |
| `deaths` | INTEGER | DEFAULT 0 | Кількість летальних випадків |
| `updated_at` | TEXT | DEFAULT CURRENT_TIMESTAMP | Мітка часу останнього оновлення запису |

### 3. Таблиця `myths` (Реєстр верифікації інформації)
Забезпечує контент для модуля боротьби з дезінформацією.
| Назва поля | Тип даних | Обмеження (Constraints) | Опис призначення |
| :--- | :--- | :--- | :--- |
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | Ідентифікатор міфу |
| `statement` | TEXT | NOT NULL | Сформульоване фейкове твердження або чутка |
| `refutation` | TEXT | NOT NULL | Академічне спростування з науковим обґрунтуванням |
| `source_url` | TEXT | NOT NULL | Пряме гіперпосилання на офіційну статтю (WHO/CDC) |

## 🚀 Інструкція з розгортання та запуску

### Крок 1: Клонування репозиторію
Стягніть вихідний код проєкту з репозиторію на локальну робочу станцію:
```bash
git clone [https://github.com/annachentoroh/epiwatch.git](https://github.com/annachentoroh/epiwatch.git)
cd epiwatch

```

### Крок 2: Встановлення залежностей та налаштування сервера (Backend)

Перейдіть у директорію бекенду та виконайте завантаження необхідних бінарних модулів з npm-реєстру:

```bash
cd backend
npm install

```

*Пакет менеджер автоматично встановить `express`, `sqlite3` та `cors` згідно з дескриптором `package.json`.*

### Крок 3: Первинне наповнення БД та запуск сервера

Перед запуском ви можете виконати первинний збір даних через скрапер (опціонально):

```bash
node scraper.js

```

Ініціалізуйте запуск вебсервера Express:

```bash
node server.js

```

*За замовчуванням HTTP-сервер почне слухати вхідні з'єднання на порту `3000`. У консолі з'явиться інформаційне повідомлення: `Server running on http://localhost:3000`.*

### Крок 4: Запуск інтерфейсу користувача (Frontend)

Оскільки клієнтська частина реалізована за технологією Vanilla JS без потреби у компіляції (компілятори/бандлери відсутні), ви можете відкрити файл `frontend/index.html` напряму через браузер.

Проте, задля уникнення обмежень безпеки політики CORS при роботі з локального диска (`file://`), рекомендується запускати фронтенд за допомогою локального статичного вебсервера:

* **Варіант А (VS Code):** Використовуйте розширення **Live Server**. Натисніть правою кнопкою миші на `index.html` -> *Open with Live Server*.
* **Варіант Б (Node.js утиліта `serve`):**
```bash
cd ../frontend
npx serve .

```


Клієнтський інтерфейс буде доступний за адресою `http://localhost:5000` або `http://localhost:3000` (залежно від конфігурації утиліти).

### ☁️ Розгортання на Production (Render Deployment)

Проєкт повністю підготовлений до автоматичного хмарного деплою на сервісі **Render**:

1. **Бекенд (Web Service):** Створюється новий Web Service, прив'язується до репозиторію. Environment: `Node`, Build Command: `cd backend && npm install`, Start Command: `node backend/server.js`.
2. **Фронтенд (Static Site):** Створюється Static Site. Publish directory: `frontend`. В конфігураційному файлі `frontend/js/app.js` перемінна `API_BASE_URL` перемикається з `http://localhost:3000` на URL-адресу задеплоєного вище Web Service.

## 📡 Документація REST API ендпоінтів

Взаємодія між клієнтом і сервером відбувається за стандартом JSON. Нижче наведено специфікацію відкритих REST-маршрутів бекенду:

### 🦕 1. Каталог захворювань

* **`GET /api/diseases`**
* **Опис:** Повертає повний масив зареєстрованих хвороб із їхніми параметрами фільтрації та рівнями загрози.
* **Формат відповіді (200 OK):**
```json
[
  {
    "id": 1,
    "name": "COVID-19",
    "type": "Вірусне",
    "description": "Гостра респіраторна інфекція...",
    "incubation_period": "2-14 днів",
    "danger_level": "High"
  }
]

```




* **`GET /api/diseases/:id`**
* **Опис:** Повертає детальну інформаційну картку хвороби за її ідентифікатором.



### 📊 2. Епідеміологічна статистика та Геодані

* **`GET /api/epidemiology/summary`**
* **Опис:** Надає масив просторових точок із географічними координатами та агрегованими даними спалахів для передачі безпосередньо в об'єкт карти `L.marker()`.
* **Формат відповіді (200 OK):**
```json
[
  {
    "disease_name": "COVID-19",
    "region": "Київська область",
    "latitude": 50.4501,
    "longitude": 30.5234,
    "confirmed_cases": 12450,
    "recovered": 11200,
    "deaths": 250,
    "danger_level": "High"
  }
]

```




* **`GET /api/epidemiology/timeline?disease_id=:id`**
* **Опис:** Повертає історичну хронологію змін кількості інфікованих для побудови таймлайнів та оновлення масивів даних `datasets` у `Chart.js`.



### 🔍 3. Скринінг та Верифікація

* **`GET /api/myths`**
* **Опис:** Запит на отримання повного списку задокументованих медичних міфів, наукових спростувань та джерел.


* **`POST /api/screening/analyze`**
* **Опис:** Надсилає обраний користувачем масив симптомів.
* **Тіло запиту (Request Body):** `{"symptoms": ["лихоманка", "сухий кашель"]}`
* **Формат відповіді (200 OK):** `{"match_probability": 85, "suggested_diseases": ["COVID-19", "Грип А"]}`




## 💻 Інструкція користувача та опис інтерфейсу

### 1. Епідеміологічний Дашборд (Головна сторінка)

При переході на `index.html` завантажується головний екран системи. Панель навігації (Header) містить логотип та посилання на всі функціональні модулі платформи. На дашборді виводяться сумарні картки ключових показників (Total Cases, Active Outbreaks, Verified Myths) та динамічний рядок глобального пошуку хвороб із системою автодоповнення тексту.

### 2. Навігація по інтерактивній карті (`map.html`)

* Система автоматично ініціалізує карту, центровану за географічними координатами України.
* Користувач може масштабувати карту колесом миші (Scroll Zoom) або кнопками керування у верхньому лівому кутку інтерфейсу.
* Натискання на будь-який кольоровий маркер активує спливаюче вікно Leaflet Popup. Попап містить структуровану таблицю з поточними статистичними показниками регіону. Через панель фільтрів карти можна відсіяти маркери, відобразивши лише хвороби з високим рівнем загрози або конкретний вид збудника (наприклад, тільки бактеріальні інфекції).

### 3. Перегляд аналітичних графіків (`charts.html`)

В інтерфейсі аналітики розташовано селектор (Dropdown) для вибору нозології. Після вибору хвороби JavaScript-модуль `charts.js` робить асинхронний запит до API, отримує хронологічні дані та ініціалізує об'єкт `new Chart()`. Користувач може перемикати часові інтервали (Тиждень / Місяць / Рік) та наводити курсор на точки графіків для виклику інструментальних підказок (Tooltips) з точними цифрами на конкретну дату.

### 4. Онлайн-тестування симптомів (`test.html`)

Користувачеві пропонується список поширених симптомів, організований у вигляді інтерактивної форми з чекбоксами. Для проведення скринінгу необхідно:

1. Зазначити галочками наявні симптоми.
2. Натиснути екранну кнопку **«Аналізувати симптоми»**.
3. Система виконає розрахунок і відобразить діалогове вікно з результатом. Користувачеві надаються базові рекомендації щодо першої допомоги та профілактики відповідно до виявленої потенційної загрози.
